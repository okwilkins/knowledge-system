## 18.2 — Introduction to iterators

### Standard library iterators

- Iterating is such a common operation, that all standard library container offer direct support for iteration.
- Instead of manually calculating the begin and end points, the container can be asked for its begin and end points.
	- This is done via member functions named `begin()` and `end()`.


```cpp
#include <array>
#include <iostream>

int main() {
	std::array array{1, 2, 3};
	
	auto begin{array.begin()};
	auto end{array.end()};
	
	for (auto p{begin}; p != end; ++p) {
		std::cout << *p << ' ';	
	}
	
	std::cout << '\n';
	
	return 0;
}
```

- The `iterator` header also contains two generic functions (`std::begin` and `std::end`) that can be used.

```cpp
#include <array>
#include <iostream>

int main() {
	std::array array{1, 2, 3};
	
	auto begin{std::begin(array)};
	auto end{std::end(array)};
	
	for (auto p{begin}; p != end; ++p) {
		std::cout << *p << ' ';
	}
	
	std::cout << '\n';
	
	return 0;
}
```

### Back to range-based for loops

- All types that implement `begin()` and `end()` member functions or can be used with `std::begin()` and `std::end()` are usable in range-based for-loops.

```cpp
#include <array>
#include <iostream>

int main() {
	std::array array{1, 2, 3};
	
	for (int i : array) {
		std::cout << i << ' ';	
	}
	
	std::cout << '\n';
	
	return 0;
}
```

- Behind the scenes, the range-based for-loop calls `begin()` and `end()` of the type to iterator over.
- For example, C-style fixed arrays can be used with `std::begin()` and `std::end()` functions.
	- So this can be used with a range-based for loop.
- Dynamic C-style arrays (or decayed C-style arrays) don't work though.
	- This is because there is no `std::end()` function for them.
		- The type information doesn't contain the array's length.
- Range-based for loops aren't the only thing that makes use of iterators.
	- They are also used in `std::sort` and other algorithms.

### Iterator invalidation (dangling iterators)

- Much like pointers and references, iterators can be left dangling.
	- This happens when the elements being iterates over change address or are destroyed.
	- This is to say that the iterator has been **invalidated**.
- Some operations that modify containers, such as adding an element to a `std::vector`, can have the side effect of causing the elements in the container to change addresses.
	- When this happens, existing iterators to those elements will be invalidated.
- Good C++ reference docs should note which container operations may or will invalidate iterators.

- Since range-based for-loops use iterators behind the scenes, caution must be taken to not do anything that invalidates the iterators of the container that is being actively traversed:
```cpp
#include <vector>

int main() {
	std::vector v{0, 1, 2, 3};
	
	for (auto num : v) {
		if (num % 2 == 0) {
			// When this invalidates the iterators of v,
			// undefined behaviour will result
			v.push_back(num + 1);
		}	
	}
	
	return 0;
}
```

- Here's another example of iterator invalidation:

```cpp
#include <iostream>
#include <vector>

int main() {
	std::vector v{1, 2, 3, 4, 5, 6, 7};
	
	auto it{v.begin()};
	++it;
	std::cout << *it << '\n';

	// Erase the element currently being iterated over
	v.erase(it);
	
	// erase() invalidates iterators to the erased
	// element (and subsequent elements)
	++it;
	// Undefined behaviour
	std::cout << *it << '\n';
	
	return 0;
}
```

- Invalidated iterators can be re-validated by assigning a valid iterator to them (e.g. `begin()`, `end()` or the return valid of some other function that returns an iterator).

- The `erase()` function returns an iterator to the element one past the erased element (or `end()` if the last element was removed).
- Therefore, the above code can be fixed:

```cpp
#include <iostream>
#include <vector>

int main() {
	std::vector v{1, 2, 3, 4, 5, 6, 7};p
	
	auto it{v.begin()};
	++it;
	std::cout << *it << '\n';

	// Erase the element currently being iterated over
	// set it to next element
	it = v.erase(it);
	
	std::cout << *it << '\n';
	
	return 0;
}
```
