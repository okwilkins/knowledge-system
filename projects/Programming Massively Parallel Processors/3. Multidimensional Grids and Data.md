## 3.1 Multidimensional Grid Organisation

- In CUDA, all threads in a grid execute the same kernel function.
- They rely on coordinates to distinguish themselves from each other.
	- And to identify the appropriate portion of the data to processes.
- A grid consists of one or more blocks.
	- Each block consists of one or more threads.

- This code will generate a 1D grid, that consists of 32 blocks, each of which consists of 128 threads.
	- Number of threads is `128 * 32 = 4096`.

```c
dim3 dimGrid(32, 1, 1);
dim3 dimBlock(128, 1, 1);
vecAddKernel<<<dimGrid, dimBlock>>(...);
```

- The grid and block dimension can also be calculated from other variables.

```c
dim3 dimGrid(ceil(n / 256.0), 1, 1);
dim3 dimBlock(256, 1, 1);
vecAddKernel<<<dimGrid, dimBlock>>>(...);
```

- For convenience, CUDA provides a special shortcut for calling a kernel with 1D grids and blocks.

```c
vecAddKernel<<<ceil(n / 256.0), 256>>>(...);
```

- This is similar to how C++ allows for default parameters.
	- C does not have this ability.
- The other parameters will take a default value of 1.
- Within the kernel function, the x field variables `gridDim` and `blockDim` are preinitialised according to the values of the execution configuration parameters.
	- For example, if n is 4000, references to `gridDim.x` and `blockDim.x` in `vectAddkernel` will result in 16 and 256 respectively.
		- Not that unlike `dim3` variables, `gridDim` and `blockDim` are built-in variables and cannot be changed.
- The total size of a block in current CUDA systems is limited to 1024 threads.
	- The total size of a block in current CUDA systems is limited to 1024 threads.
	- These threads can be distributed across the three dimensions in any way, as long as the total number of threads does not exceed 1024.


## 3.2 Mapping Threads to Multidimensional Data

- The choice of 1D, 2D or 3D thread organisations is usually based on the nature of the data.
	- For example, pictures are a 2D array of pixels.
- Dimensions of multidimensional data will be refereed to in descending order.
	- For example 62 X 76 picture is a 76 wide and 62 height picture.
	- Also  this will be reflected in this notation here:  $$P_{y,x}$$	

- Ideally, it would be nice to access a picture as a 2D array like this: `Pin_d[j][i]`.
- However, the number of columns in `Pin` need to be known at compile time for `Pin` to be accessed as a 2D array.
	- This info is not known at compile time for dynamically allocated arrays.
	- The info on the number of columns in a dynamically allocated 2D array is not known at compile time by design.
	- As a result, programmers need to explicitly linearise or "flatten" a dynamically allocated 2D array into an equivalent 2D array in the current CUDA C.
- In reality, all multidimensional arrays in C are linearised.
	- Under the hood, the compiler does this automatically.
	- In the case of dynamically allocated arrays, the current CUDA c compiler leaves he work of such translation to the programmers, owing to lack of dimensional info at compile time.


- Example of converting an image to greyscale:

```c
// The input image is encoded as unsigned chars [0, 255]
// Each pixel is 3 consecutive chars for the 3 channels (RGB)
__global__ void colourToGreyscaleConvertion(
	unsigned char *Pout,
	unsigned char *Pin,
	int width,
	int height,
) {
	int col = blockIdx.x*blockDim.x + threadIdx.x;
	int row = blockIdx.y*blockDim.y + threadIdx.y;
	
	if (col >= width || row >= height) {
		return;	
	}

	// Get 1D offset for the greyscale image
	int greyOffset = row*width + col;
	// Can think of RGB image having CHANNEL
	// times more columns than the grewscale image
	int rgbOffset = greyOffset * CHANNELS
	unsigned char r = Pin[rgbOffset];
	unsigned char g = Pin[rgbOffset + 1];
	unsigned char b = Pin[rgbOffset + 2];
	
	// Perform the rescaling and store it
	// Multiply by floating point constants
	Pout[greyOffset] = 0.21f*r + 0.71*g + 0.07*b;
}
```


## 3.3 Image Blur: A More Complex Kernel

```c
__global__ void blurKernel(
	unsigned char *in,
	unsigned char *out,
	int w,
	int h,
) {
	int col = blockIdx.x*blockDim.x + threadIdx.x;
	int row = blockIdx.y*blockDim.y + threadIdx.y;

	if (col >= w || row >= h) {
		return;	
	}
	
	int pixVal = 0;
	int pixels = 0;	
	
	// Get average of the surrounding BLUR_SIZE x BLUR_SIZE box
	for (int blurRow=-BLUR_SIZE; blurRow < BLUR_SIZE + 1; ++blurRow) {
		for (int blurCol=-BLUR_SIZE; blurCol < BLUR_SIZE + 1; ++blurCol) {
			int curRow = row + blurRow;
			int curCol = col + blurCol;
	
			// Verify  is a valid image pixel
			if (curRow >= 0 && curRow < h && curCol >= 0 && curCol < w) {
				pixVal += in[curRow*w + curCol];
				// Keep track of number of pixels in the avg
				++pixels;
			}
		}
	}	

	out[row*w + col] = (unsigned char)(pixVal / pixels);
}
```


## 3.4 Matrix Multiplication

- The inner product of the individual vector elements is: $$P_{row,col}=\sum{M_{row,i}}*N_{i,col} \text{ for i=0, 1, 2..., width - 1}$$

```c
__global__ void matMulKernel(
	// Matrix 1
	float *M,
	// Matrix 2
	float *N,
	// Product matrix
	float *P,
	// Width of both the M/N matries that are AxA matries
	int matSize,
) {
	int row = blockIdx.x*blockDim.x + threadIdx.x;
	int col = blockIdx.y*blockDim.y + threadIdx.y;
	
	if ((row >= matSize) || (col >= matSize)) {
		return;	
	}
	
	float out = 0;

	for (int i = 0; i < matSize; ++i) {
		out += M[row*matSize + i] * N[i*matSize + col];
	}	
	
	P[row*matSize + col] = out;
}
```
